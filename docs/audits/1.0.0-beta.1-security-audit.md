## Methodology

The audit runs in 5 phases (see #255 for the full plan):

1. **Phase 0** — scaffolding (this commit): branch, report skeleton, triage ledger, label taxonomy.
2. **Phase 1** — automated tooling pass: `detekt`, `spotbugs + find-sec-bugs`, OWASP Dependency-Check, `eslint-plugin-security`, `npm audit`, `knip`, `gitleaks`, Trivy (vuln+secret+misconfig+license), `hadolint`. Raw output archived under [`1.0.0-beta.1-artifacts/`](./1.0.0-beta.1-artifacts/).
3. **Phase 2** — 6 parallel agent-led review lanes (`kotlin-reviewer`, `typescript-reviewer`, `database-reviewer`, `security-reviewer`, `refactor-cleaner`, `java-reviewer` loaded with the `springboot-security` skill). Each agent appends findings to [`1.0.0-beta.1-triage.csv`](./1.0.0-beta.1-triage.csv) with its ID prefix.
4. **Phase 3** — dedupe, severity calibration, follow-up issue creation, consolidated report.
5. **Phase 4/5** — in-line fixes for `critical` findings; sign-off and closure.

## Triage Process

1. Each agent appends rows to [`1.0.0-beta.1-triage.csv`](./1.0.0-beta.1-triage.csv) with its ID prefix.
2. Phase 3 dedupe: group by `file:line`; keep the owning lane's entry, cross-link others.
3. Severity calibration: `security-reviewer` agent re-runs over all `critical`/`high` rows and adjusts.
4. Follow-up issue per finding (non-critical) via:

```bash
gh issue create \
  --repo plugwerk/plugwerk \
  --title "audit(beta1): [<severity>][<scope>] <short title>" \
  --label "security-audit,beta1,severity:<sev>,<scope-label>" \
  --milestone "1.0.0-beta.1" \
  --body "<reference finding ID, audit-report anchor, file:line, recommendation; 'Part of #255'>"
```

Scope labels (reuse existing where possible):

| Scope area | Label(s) |
|---|---|
| Backend Kotlin | `backend` |
| Frontend React/TS | `frontend` |
| Persistence / Liquibase | `database` |
| PF4J client plugin | `scope:client` |
| Build / CI / Gradle / workflows | `scope:build` |
| Docker / ops | `scope:ops` |
| Documentation | `documentation` |

All audit-generated follow-up issues additionally carry `security-audit` and `beta1`.

5. `critical` findings are fixed in-line on dedicated `fix/<issue>_<short>` branches off `main` — not on this audit branch. Each critical fix must be driven by the `tdd-guide` agent (failing regression test first) and pass full `./gradlew build` / `npm run build && npm run test:run`.
6. Findings table below links each row to the follow-up issue or fix PR.

### Severity definitions

| Severity | Meaning | Gate |
|---|---|---|
| critical | Remote exploit, data loss, privilege escalation, credential exposure | Blocks 1.0.0-beta.1; fixed in-line |
| high | Significant risk with exploitable path or material quality issue | Fix before beta OR explicit accepted-risk entry below |
| medium | Latent risk or maintainability concern | Follow-up issue on beta.1 milestone |
| low | Style, minor smell | Follow-up issue, no beta gate |
| info | Observation worth recording, no action needed | Listed here, no issue |

## Limitations

Explicitly documented so readers understand what this audit does **not** cover:

- **PF4J classloader boundary is not statically analysable.** Dynamic plugin loading via `pluginClassLoader.loadClass(...)` cannot be fully traced by the tools used here. A dedicated runtime test with a hostile-descriptor fixture is tracked as a follow-up issue rather than an audit blocker.
- **Third-party OIDC IdPs are not pen-tested.** The audit verifies that Plugwerk uses OIDC correctly (issuer/audience/nonce/PKCE) but does not assess the security of external identity providers.
- **No live penetration test.** Static analysis + targeted manual review only; a dynamic pen-test is a follow-up activity post-beta.
- **Historical secrets in git history are detected (gitleaks) but not rotated here.** Rotation actions are filed as individual issues against infra.
