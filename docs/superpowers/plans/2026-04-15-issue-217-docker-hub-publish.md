# Issue #217 — Publish Plugwerk Server to Docker Hub Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Publish the Plugwerk Server as a multi-platform Docker image (`linux/amd64` + `linux/arm64`) to Docker Hub as `plugwerk/plugwerk-server:<version>` on every `v*` tag push.

**Architecture:** Extend `.github/workflows/release.yml` with (a) a new step in the existing `publish` job that uploads the freshly built backend fat JAR as a short-lived workflow artifact and (b) a new `docker-publish` job with `needs: publish` that downloads the JAR, stages it next to the existing `Dockerfile` as Docker build context, and runs `docker/build-push-action@v6` via `buildx` + QEMU. Image tags are generated by `docker/metadata-action@v5` — `{{version}}` always, and `latest` only when the tag name contains no `-` (i.e. is a stable release). The existing `src/dist/Dockerfile` stays unchanged; no new Dockerfile is introduced. The distribution `docker-compose.yml` is switched from `build: .` to reference the published image so extracted distributions pull from Docker Hub. Documentation in `docs/deployment.md` gains a new top-level option describing `docker pull plugwerk/plugwerk-server:<version>`.

**Tech Stack:** GitHub Actions, `docker/setup-qemu-action@v3`, `docker/setup-buildx-action@v3`, `docker/login-action@v3`, `docker/metadata-action@v5`, `docker/build-push-action@v6`, `actions/upload-artifact@v4`, `actions/download-artifact@v4`, Docker Buildx, `eclipse-temurin:21-jre-alpine` base image (unchanged).

**Registry decision:** Docker Hub only (GHCR explicitly not included per user decision 2026-04-15).

**Required repository secrets** (must be added manually by a repo admin before the workflow runs; document in PR description):
- `DOCKERHUB_USERNAME` — Docker Hub account name with push rights on `plugwerk/plugwerk-server`
- `DOCKERHUB_TOKEN` — Docker Hub Personal Access Token (scope: Read & Write), **not** the account password

## Task 2: Add `docker-publish` job skeleton (build only, no push)

**Why:** Land the job wiring — checkout, download artifact, stage JAR, set up buildx/QEMU, run a local-only `buildx build` — without any external side effects. This proves the Dockerfile accepts the staged JAR and the build succeeds for both architectures before we enable registry push.

**Files:**
- Modify: `.github/workflows/release.yml` (append new job after `publish`)

- [ ] **Step 1: Append the `docker-publish` job**

Append to the end of `.github/workflows/release.yml` (as a sibling of the `publish` job under `jobs:`):

```yaml
  docker-publish:
    name: Build and push Docker image
    needs: publish
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Extract version from tag
        id: version
        run: echo "VERSION=${GITHUB_REF_NAME#v}" >> "$GITHUB_OUTPUT"

      - name: Download backend fat JAR from publish job
        uses: actions/download-artifact@v4
        with:
          name: plugwerk-server-backend-jar
          path: plugwerk-server/plugwerk-server-backend/src/dist/

      - name: Verify staged JAR is present
        run: |
          STAGED="plugwerk-server/plugwerk-server-backend/src/dist/plugwerk-server-backend-${{ steps.version.outputs.VERSION }}.jar"
          ls -lh "$STAGED"
          test -f "$STAGED" || { echo "::error::Staged JAR missing: $STAGED"; exit 1; }

      - name: Set up QEMU
        uses: docker/setup-qemu-action@v3

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build image (no push, smoke test both platforms)
        uses: docker/build-push-action@v6
        with:
          context: plugwerk-server/plugwerk-server-backend/src/dist
          file: plugwerk-server/plugwerk-server-backend/src/dist/Dockerfile
          platforms: linux/amd64,linux/arm64
          push: false
          tags: plugwerk/plugwerk-server:ci-smoke
```

- [ ] **Step 2: Validate workflow YAML syntax**

```bash

actionlint .github/workflows/release.yml 2>/dev/null || echo "actionlint not installed (ok)"
```
Expected: `YAML OK`.

- [ ] **Step 3: Local Dockerfile build validation (linux/amd64 only, native)**

Build the fat JAR once locally, then run a native `buildx build` mimicking what CI does (skip `linux/arm64` locally — QEMU emulation is slow and the native arch is enough to validate Dockerfile + staging logic).

